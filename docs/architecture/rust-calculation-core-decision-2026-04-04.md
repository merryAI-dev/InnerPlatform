# Rust 계산 코어 도입 의사결정

작성일: 2026-04-04  
대상: CTO / 임원진 / 제품 / 플랫폼 엔지니어링

## 결정 요약

- 제품 전체를 Rust로 재작성하지 않는다.
- 대신 `정산/예산/cashflow 계산 코어`를 Rust로 분리하는 방향을 전략안으로 채택한다.
- UI, 워크플로우, 운영 화면은 React/TypeScript를 유지한다.

한 줄 요약:

> 전사 1,000명 규모에서 병목은 “화면 기술”이 아니라 “시트 규칙을 대량 데이터와 동시성 환경에서 얼마나 빠르고 일관되게 계산하느냐”이므로, 계산 커널만 Rust로 재구성하는 것이 가장 투자 대비 효과가 크다.

## 왜 지금 이 판단이 필요한가

- 시트를 많이 활용하는 팀일수록 지연에 대한 허용도가 매우 낮다.
- PM은 “저장 후 다시 계산되며 값이 바뀌는 경험”을 신뢰 훼손으로 받아들인다.
- 정산/예산/cashflow는 한 화면이 아니라 서로 연결된 계산 체인이라, 한 군데의 지연이나 불일치가 전체 운영 신뢰를 무너뜨린다.
- 사용자가 늘수록 “느림”보다 더 위험한 것은 “같은 입력인데 사람마다 다른 결과를 보는 것”이다.

즉, 이 문제는 프론트엔드 최적화의 하위 이슈가 아니라 전사 운영 시스템의 핵심 계산 아키텍처 문제다.

## 왜 Rust인가

### 1. 결정적 계산

정산 시스템은 입력이 같으면 결과가 항상 같아야 한다.

- 행 파생 계산
- 잔액 흐름
- 예산 소진 집계
- 주차별 cashflow actual/projection 계산
- 검증 규칙 적용

Rust는 계산 코어를 순수 함수 중심으로 강하게 고정하기 좋다.

### 2. 대량 처리와 메모리 예측성

지출대장/정산대장/예산/cashflow가 연결되면 한 번의 편집이 여러 파생 결과를 유발한다.

- 전체 재계산
- 부분 재계산
- 검증기
- 제출 전 리뷰 패널

이 경로를 JS 상태 트리 안에서만 모두 처리하면, 데이터가 커질수록 성능과 디버깅 비용이 빠르게 나빠진다.

### 3. 서버와 브라우저 재사용

Rust 계산 엔진은 다음 두 경로에서 재사용 가치가 높다.

- 서버 authoritative 계산
- 브라우저 WASM preview 계산

즉, 저장 전 미리보기와 저장 후 확정 계산이 같은 코어를 공유할 수 있다.

## 왜 전면 Rust 재작성은 하지 않는가

### 1. 병목의 80%는 계산 경로다

현재 핵심 문제는 다음이다.

- 대량 행 편집 시 재계산 지연
- 시트식 규칙의 불투명성
- 저장/동기화 시 결과 복원 또는 변경
- 동시 수정/재계산 충돌 가능성

이것은 화면 렌더링보다 계산 경로와 상태 일관성 문제다.

### 2. UI는 React/TypeScript가 더 빠르게 진화한다

운영 화면은 계속 바뀐다.

- PM UX 피드백 반영
- 배지/패널/가이드 구조 변경
- 검토 패널/에러 표시/탭 UX 개선

이 반복은 React/TypeScript가 훨씬 빠르고 조직적으로 유리하다.

## 목표 아키텍처

- DB가 source of truth
- Google Sheet는 import/export 채널
- 서버가 최종 authoritative 계산 주체
- 브라우저는 편집과 preview 담당
- 계산 규칙은 수식 문자열이 아니라 명시적 도메인 규칙으로 관리

구조:

1. `React/TypeScript UI`
2. `Rust calculation core`
3. `Server authoritative layer`
4. `WASM preview layer`

## 실행 순서

1. 워크북 전체 수식/탭 기준선 고정
2. TypeScript 기준 authoritative engine 완성
3. dual-run parity 확보
4. 계산 코어만 Rust로 이관
5. 전사 cutover 후 앱을 공식 source of truth로 전환

## 비고

이번 결정은 “기술 취향”이 아니라 다음을 위한 결정이다.

- 전사 규모에서의 신뢰성
- 예측 가능한 응답성
- 동일 입력에 대한 동일한 결과
- 브라우저/서버 계산 불일치 제거

이 기준에서는 Rust 계산 코어 전략이 가장 타당하다.
