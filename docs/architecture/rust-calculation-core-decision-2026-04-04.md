# Rust 계산 코어 도입 의사결정

작성일: 2026-04-04  
대상: CTO / 임원진 / 제품 / 플랫폼 엔지니어링

## 1. 결정 요약

결정:

- 제품 전체를 Rust로 재작성하지 않는다.
- 대신 `정산/예산/cashflow 계산 코어`를 Rust로 분리하는 방향을 전략안으로 채택한다.
- UI, 워크플로우, 운영 화면은 React/TypeScript를 유지한다.

한 줄 요약:

> 전사 1,000명 규모에서 병목은 “화면 기술”이 아니라 “시트 규칙을 대량 데이터와 동시성 환경에서 얼마나 빠르고 일관되게 계산하느냐”이므로, 계산 커널만 Rust로 재구성하는 것이 가장 투자 대비 효과가 크다.

## 2. 왜 지금 이 판단이 필요한가

현재 사용자 불만의 본질은 단순 미관 문제가 아니다.

- 시트를 많이 활용하는 팀일수록 지연에 대한 허용도가 매우 낮다.
- PM은 “저장 후 다시 계산되며 값이 바뀌는 경험”을 신뢰 훼손으로 받아들인다.
- 정산/예산/cashflow는 한 화면이 아니라 서로 연결된 계산 체인이라, 한 군데의 지연이나 불일치가 전체 운영 신뢰를 무너뜨린다.
- 사용자가 늘수록 “느림”보다 더 위험한 것은 “같은 입력인데 사람마다 다른 결과를 보는 것”이다.

즉, 이 문제는 프론트엔드 최적화의 하위 이슈가 아니라 전사 운영 시스템의 핵심 계산 아키텍처 문제다.

## 3. 왜 Rust인가

Rust를 선택하는 이유는 속도 하나가 아니다.

### 3.1 결정적 계산

정산 시스템은 입력이 같으면 결과가 항상 같아야 한다.

- 행 파생 계산
- 잔액 흐름
- 예산 소진 집계
- 주차별 cashflow actual/projection 계산
- 검증 규칙 적용

이 계산이 브라우저마다, 구현체마다, 상태 타이밍마다 흔들리면 전사 시스템으로 쓸 수 없다.

Rust는 계산 코어를 순수 함수 중심으로 강하게 고정하기 좋다.

### 3.2 메모리 예측성과 대량 처리

지출대장/정산대장/예산/cashflow가 연결되면 한 번의 편집이 여러 파생 결과를 유발한다.

- 전체 재계산
- 부분 재계산
- 검증기
- 제출 전 리뷰 패널

이 경로를 JS 상태 트리 안에서만 모두 처리하면, 데이터가 커질수록 성능과 디버깅 비용이 빠르게 나빠진다.

Rust는 대량 row 처리, 집계, 검증을 예측 가능한 메모리/CPU 모델로 설계하기에 유리하다.

### 3.3 서버와 브라우저의 동일 엔진 재사용

Rust 계산 엔진은 다음 두 경로에서 재사용 가치가 높다.

- 서버 authoritative 계산
- 브라우저 WASM preview 계산

즉, 저장 전 미리보기와 저장 후 확정 계산이 같은 코어를 공유할 수 있다.

## 4. 왜 전면 Rust 재작성은 하지 않는가

전면 Rust 재작성은 기술적으로 가능해도 경영적으로는 비효율적이다.

### 4.1 병목의 80%는 계산 경로다

현재 핵심 문제는:

- 대량 행 편집 시 재계산 지연
- 시트식 규칙의 불투명성
- 저장/동기화 시 결과 복원 또는 변경
- 동시 수정/재계산 충돌 가능성

이것은 화면 렌더링보다 계산 경로와 상태 일관성 문제다.

### 4.2 UI는 React/TypeScript가 더 빠르게 진화한다

운영 화면은 계속 바뀐다.

- PM UX 피드백 반영
- 배지/패널/가이드 구조 변경
- 검토 패널/에러 표시/탭 UX 개선

이 반복은 React/TypeScript가 훨씬 빠르고 조직적으로 유리하다.

### 4.3 채용과 유지보수 비용

전면 Rust는 엔지니어링 우아함보다 조직 비용이 더 커질 가능성이 높다.

따라서 “계산 코어는 Rust, 제품 UI는 TypeScript”가 현실적인 최적점이다.

## 5. 목표 아키텍처

### 5.1 원칙

- DB가 source of truth
- Google Sheet는 import/export 채널
- 서버가 최종 authoritative 계산 주체
- 브라우저는 편집과 preview 담당
- 계산 규칙은 수식 문자열이 아니라 명시적 도메인 규칙으로 관리

### 5.2 구조

1. `React/TypeScript UI`
- 표 편집
- 상태 표시
- 검토 패널
- 선택 영역 액션

2. `Rust calculation core`
- 행 파생 계산
- 잔액 흐름
- 예산 소진 집계
- 주차별 actual/projection 집계
- 정책/검증 규칙 실행

3. `Server authoritative layer`
- 저장 시 최종 계산
- 충돌/검증 처리
- 이벤트 로그 또는 append-only ledger 기반 확장 가능

4. `WASM preview layer`
- 셀 편집 직후 빠른 재계산
- 서버 저장 전 결과 미리보기

## 6. Rust로 분리할 대상

1차 우선 대상:

- `사용내역` row derivation
- running balance 계산
- `매입부가세/공급가액/통장금액` 보조 계산
- 주차별 cashflow actual 집계
- 예산총괄시트 소진/부가세 합산
- 정책 기반 validation

2차 대상:

- 대량 import normalize
- diff/reconcile 엔진
- 제출 전 검토 패널 집계

비대상:

- 포털 라우팅
- 일반 CRUD 화면
- UI 상태/디자인 시스템

## 7. 성공 기준

기술적 성공 기준은 다음처럼 잡는다.

- 편집 후 preview 재계산 P95 < 100ms
- 서버 저장 후 authoritative recompute P95 < 300ms
- 1,000행 이상에서도 부분 재계산 체감 지연 최소화
- 동일 입력에 대해 브라우저 preview와 서버 확정 계산 결과가 일치
- Google Sheet 기준 golden test 통과율 100%

## 8. 선행 조건

Rust로 바로 넘어가기 전에 반드시 해야 할 것이 있다.

### 8.1 시트 수식 parity 고정

먼저 해야 할 일:

- Google Sheet 수식 인벤토리 추출
- 핵심 시트 의존성 정리
- 계산 규칙 분해
- golden test 작성

우리가 이미 확인한 핵심 체인은 다음과 같다.

- `사용내역(통장내역기준취소내역,불인정포함)`  
  행 단위 잔액/사업비 사용액/매입부가세 파생
- `예산총괄시트`  
  사용내역 기반 소진금액/매입부가세 집계
- `cashflow(사용내역 연동)`  
  사용내역 기반 주차별 actual 집계

즉, Rust 전환의 출발점은 “더 빠른 구현”이 아니라 “현재 시트 의미를 정확히 고정하는 것”이어야 한다.

### 8.2 깨진 시트 수식 정리

원본 시트 내 `#REF!` 수식도 일부 확인되었다.

이 의미는 중요하다.

- 원본 시트 자체가 완전한 정답은 아니다.
- 따라서 앱은 “시트를 그대로 복제”가 아니라 “검증된 규칙을 도메인 엔진으로 정제”해야 한다.

## 9. 단계별 실행안

### Phase 0. 수식 인벤토리 고정

- Google Sheet 수식 전수 추출
- 핵심 시트별 의존성 맵 작성
- golden test 샘플셋 구축

### Phase 1. TypeScript 기준 도메인 규칙 고정

- 현재 앱에서 계산 규칙 정리
- 시트 parity 확보
- 브라우저와 서버의 계산 일관성 확보

이 단계가 필요한 이유:

> Rust로 바로 옮기면 “빠르지만 틀린 엔진”을 만들 위험이 있기 때문이다.

### Phase 2. Rust calculation core 도입

- 순수 계산 함수 이관
- 테스트셋 그대로 재사용
- 서버 authoritative 계산에 우선 투입

### Phase 3. WASM preview 도입

- 브라우저 입력 즉시성 개선
- 부분 재계산 최적화

### Phase 4. 고급 운영 기능

- provenance 추적
- deterministic replay
- conflict-safe editing
- append-only ledger/event sourcing 검토

## 11. 최종 판단

CTO 관점 최종 판단은 다음과 같다.

- `지금 당장 전체 Rust 전환`은 하지 않는다.
- `계산 엔진을 Rust로 분리하는 장기 전략`은 채택한다.
- `수식 parity와 golden test 확보`를 Rust 전환의 선행 게이트로 둔다.

즉, 이 결정은 “기술 취향”이 아니라 다음을 위한 결정이다.

- 전사 규모에서의 신뢰성
- 예측 가능한 응답성
- 동일한 입력에 대한 동일한 결과
- 브라우저/서버 계산 불일치 제거

이 기준에서는 Rust 계산 코어가 가장 타당한 방향이다.
